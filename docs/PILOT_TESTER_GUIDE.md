# Hunter — гайд для пилотных тестеров (TZ-4 features)

> Этот документ — что и как тестировать в первые 7 дней пилота.
> Цель: собрать данные для решений по `enforce mode` + `ClientProfile cutover` + `legacy event removal`.

## 0. Контекст

Платформа находится в **warn-only режиме** для AI policy engine. Это значит:
- Все новые контракты (audit, persona memory, attachment pipeline) **активны** и пишут события в canonical event log
- Но ничто не блокирует ваш диалог с AI — даже если AI нарушает policy, реплика проходит
- Цель пилота: накопить данные о том, **сколько раз** policy engine ловит реальные нарушения, чтобы потом повернуть рубильник в `enforce mode` без шквала ложных блокировок

## 1. Маршруты для проверки

### 1.1 Тренировка из карточки клиента (CRM-сессия)

**Путь**: `/clients/[id]` → "Начать тренировку" → `chat` или `call`

**Что должно работать**:
- В правой панели ("ИСТОРИЯ ВЗАИМОДЕЙСТВИЙ") — каждый загруженный документ виден строкой с `Paperclip` иконкой + filename + чипами (verified / duplicate, если применимо)
- В левой панели "ДОКУМЕНТЫ" каждый attachment имеет 3-4 chip:
  - lifecycle (Получен / Отклонён)
  - OCR (только если документ — PDF / image)
  - classification (только если идёт)
  - verification (только если кто-то ревьюнул)
- Под "ДОКУМЕНТЫ" — секция "ПАМЯТЬ КЛИЕНТА" (если AI с вами уже общался по этому клиенту):
  - текущий MemoryPersona snapshot (имя, форма обращения «вы»/«ты»)
  - `Lock`-чипы для подтверждённых slot-фактов (город, доход, и т.д.)
  - последний SessionPersonaSnapshot
  - счётчики событий за 30 дней

**Что регистрировать (репорт в Slack)**:
- Если AI говорит про вас в третьем лице или путает имя — это `persona.conflict_detected` event, должен попасть в Методология → Качество AI
- Если AI спрашивает то же самое второй раз (например, ваш доход) — `near_repeat` или `asked_known_slot_again`
- Если документ загружается, но статус остаётся "Получен" дольше 30 секунд — write-path issue, репортнуть

### 1.2 Тренировка вне CRM (story mode)

**Путь**: `/training` → выбрать сценарий → "Звонок (история на 3-5 звонков)"

**Что должно работать**:
- Цепочка из 3-5 звонков с одним AI-клиентом (один и тот же character)
- Между звонками — экран "between_calls" с краткой сводкой
- Идентичность AI-клиента (имя, голос, gender) **должна быть одинакова во всех звонках цепочки** — это критично, ни одно из имён не должно "переключиться"

**Что регистрировать**:
- Если в звонке #2 имя AI-клиента отличается от звонка #1 — критический bug, screenshot обязательно
- Если voice goes другой — тоже репортнуть

### 1.3 Центр (одиночный звонок с фиксированным исходом)

**Путь**: `/center` → выбрать сценарий → "Начать звонок"

**Что должно работать**:
- Сессия открывается без ошибки `lead_client_required` (закрыто в C1 fix)
- В конце звонка обязательный выбор исхода: `Договор согласован` / `Договор не согласован` / `Продолжить в другом звонке`
- Без выбора исхода сессия не завершается

**Что регистрировать**:
- Если кнопка "Начать звонок" даёт 400 — деплой не докатился, репортнуть
- Если экран выбора исхода не появляется — баг completion policy

### 1.4 /home — быстрый старт с превью клиента

**Путь**: `/home` → ждёте превью AI-клиента → "Принять"

**Что должно работать**:
- Имя/город/долг которые вы видите в превью **=** именно те данные с которыми AI начнёт разговор (root cause hotfix #55)
- Если несколько вкладок открыто — каждая видит своего клиента, не перепутаются

## 2. Что РОП / admin видит дополнительно

### 2.1 Методология → "Качество AI" (новая sub-tab — D7.7a)

**Путь**: `/dashboard?tab=methodology&sub=ai_quality`

Окно фильтр: 1d / 7d / 30d. По умолчанию — 7 дней.

Показывает:
- **Топ-цифры**: сколько Policy Violations + Persona Conflicts + slot_locked за окно
- **Severity strip**: разбивка по low/medium/high/critical
- **Top codes**: какие нарушения чаще всего (`near_repeat`, `missing_next_step`, `asked_known_slot_again`...)
- **By manager**: с какого менеджера какие срабатывания идут
- **Recent feed**: последние 20 событий (severity + менеджер + session_id первые 8 символов)

**Что нужно искать**:
- За первые 7 дней должно набраться достаточно событий для FP-rate анализа
- Если какой-то код фиксирует **>20%** всех violations и при ручной проверке оказывается ложным — это red flag для enforce-flip (надо тюнить порог threshold)

### 2.2 Методология → "Ревью знаний" (D6)

**Путь**: `/dashboard?tab=methodology&sub=knowledge_review`

Очередь knowledge items в `needs_review` status. Появляется когда:
- TTL cron перевёл `actual → needs_review` (раз в день, по `expires_at < now`)
- Никогда не переходит в `outdated` автоматически — только через эту очередь

**Действия**:
- "Применить ревью" → выбор `Актуально` / `Спорно` / `Устарело` + комментарий
- `Устарело` — это терминал, элемент исключается из RAG/recommendations
- `Актуально` — возвращает в общий пул

### 2.3 Активность → AuditLogPanel

**Путь**: `/dashboard?tab=activity`

Filter "Документы" (D7.7b) — показывает все upload_attachment события по команде.

## 3. Чего НЕ делать в пилоте

- НЕ нажимайте "Применить ревью" → "Устарело" на legal_knowledge_chunks без проверки. После flip обратно "Актуально" — мгновенно, но cron не возвращает; нужно через UI.
- НЕ смешивайте свои аккаунты — каждый менеджер только своих клиентов; РОП — свою команду; admin — всех. Если вы видите чужого клиента — это RBAC bug, репортнуть.

## 4. Что точно НЕ работает (известные ограничения warn-only)

- AI policy engine **не блокирует** реплики, только логирует. Слишком длинный ответ AI всё равно отправится.
- WS push для `persona.conflict_detected` идёт в notifications WS — если у вас нет открытой вкладки, badge не обновится. После reload — счётчик восстановится из polling /me/pending-events.
- ClientProfile.full_name всё ещё используется runtime'ом для real-client сессий (пока не наберём данные SessionPersonaSnapshot — это D7.4 cutover).
- `attachment.linked` event эмитится новым pipeline'ом, но FE timeline reader всё ещё на legacy `session.attachment_linked` — это D7.5.

## 5. Как репортить

В Slack-канале #hunter-pilot, шаблон:
```
Severity: [low / medium / high / critical]
Что делал: <user flow>
Что увидел: <observed behaviour>
Что ожидал: <expected behaviour>
Session ID: <если есть>
URL: <ссылка>
Скриншот: <если применимо>
```

Critical = блокер, теряются данные. High = функционально сломано. Medium = баг видим но обходится. Low = косметика.

## 6. Что мы соберём за 7 дней warn-only window

Чтобы потом перевернуть `CONVERSATION_POLICY_ENFORCE_ENABLED=1`:
- FP rate per code (target < 5%)
- Total violations / day (target stable)
- Most-triggering manager (для coaching)
- Persona drift counter (target 0 — каждое срабатывание это bug)

После 7 дней — review meeting + flip → enforce mode.
